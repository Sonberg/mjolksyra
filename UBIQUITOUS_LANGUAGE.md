# Ubiquitous Language

_Mjolksyra — fitness coaching platform. Last updated: 2026-04-11._

---

## People

| Term        | Definition                                                                                       | Aliases to avoid               |
| ----------- | ------------------------------------------------------------------------------------------------ | ------------------------------ |
| **Coach**   | A user who manages athletes, creates workout plans, and bills for their services                 | Trainer, instructor            |
| **Athlete** | A user who follows a coach's plans, logs workouts, and pays a subscription fee                   | Client, trainee (as a person)  |
| **Trainee** | The **relationship** between a Coach and an Athlete — not the Athlete themselves                 | Athlete (when meaning the link) |
| **Admin**   | A platform operator with elevated access to all coaches, athletes, and billing data              | Superuser, staff               |

> **Key distinction:** "Athlete" refers to the person; "Trainee" refers to the coach–athlete relationship record. An Athlete can be a Trainee of multiple Coaches; each Trainee is owned by exactly one Coach.

## Workout Planning

| Term                   | Definition                                                                                                   | Aliases to avoid                       |
| ---------------------- | ------------------------------------------------------------------------------------------------------------ | -------------------------------------- |
| **Block**              | A reusable workout template created by a Coach, containing ordered workouts and exercises                    | Program, template, routine             |
| **Planned Workout**    | A single workout scheduled on a specific date for an Athlete, optionally created from a Block                | Scheduled workout, session (planned)   |
| **Exercise**           | A named movement pattern (e.g. Squat), scoped to a Coach or the platform, with type and sport metadata      | Movement, lift                         |
| **Block Exercise**     | An instance of an Exercise inside a Block, with prescribed sets/reps/duration for that Block                 | Exercise entry                         |
| **Planned Exercise**   | An instance of an Exercise inside a Planned Workout, with prescribed sets/reps/duration for that day         | Exercise entry                         |
| **Apply Block**        | The act of stamping a Block's workouts onto an Athlete's calendar as Planned Workouts                        | Copy block, import block               |
| **Draft Exercise**     | A Planned Exercise generated by the AI Planner that has not yet been published to the Athlete                | Pending exercise, unpublished exercise |
| **Publish**            | Promoting Draft Exercises to live Planned Exercises visible to the Athlete                                   | Confirm, save, approve (for exercises) |
| **Skip**               | Marking a Planned Workout as intentionally missed without deleting it                                        | Cancel workout, remove workout         |

## Relationships

- A **Coach** has zero or more **Trainees**; each **Trainee** links exactly one **Coach** to one **Athlete**
- A **Trainee** has exactly one **Trainee Cost**, which maps to exactly one **Stripe Subscription** once both parties have Stripe configured
- A **Coach** owns **Blocks**; a **Block** contains ordered **Block Workouts**, each containing **Block Exercises**
- A **Planned Workout** belongs to one **Trainee** and may have been created by applying a **Block**
- A **Planned Workout** has zero or one corresponding **Completed Workout**
- A **Completed Workout** has zero or more **Chat Messages** and zero or more **Media Analyses**
- A **Planner Session** belongs to one **Trainee** and produces one **Workout Plan Proposal** at a time
- A **Coach** has one **Credit** balance, modified by **Credit Ledger** entries; AI features consume **Credits** via **Credit Actions**

## Example dialogue

> **Dev:** "When a **Coach** wants to start billing an **Athlete**, what's the flow?"
>
> **Domain expert:** "The **Coach** sends a **Trainee Invitation**. Once the **Athlete** accepts it, a **Trainee** relationship is created. The **Coach** then sets the **Trainee Cost**, which triggers creation of a **Stripe Subscription** — but only if both sides have completed Stripe onboarding."
>
> **Dev:** "What if payment fails? Does the **Trainee** get cancelled?"
>
> **Domain expert:** "No, not immediately. A failed payment creates a **Trainee Transaction** with status `Failed`. The **Stripe Subscription** handles retries. Only a `customer.subscription.deleted` webhook cancels the **Trainee**."
>
> **Dev:** "And when a **Coach** uses the **AI Planner**, how do workouts get to the **Athlete**?"
>
> **Domain expert:** "The **Coach** starts a **Planner Session** and the AI generates a **Workout Plan Proposal** with **Draft Exercises**. The **Coach** can **Clarify** to iterate, then **Apply the AI Plan** to **Publish** the drafts — only then are **Planned Workouts** visible to the **Athlete**. If the **Coach** doesn't like it, they **Discard** the proposal."
>
> **Dev:** "Got it — so **Draft Exercises** and **Planned Exercises** are the same type, just different lifecycle states?"
>
> **Domain expert:** "Exactly. Draft is unpublished, Planned is live. The **Athlete** never sees the Draft stage."

## Flagged ambiguities

- **"Session"** is overloaded: it appears as **Workout Session** (in-progress logging state), **Planner Session** (AI conversation thread), and informally to mean **Planned Workout**. Prefer the qualified forms and avoid the bare word "session."
- **"Trainee" vs "Athlete"** are used interchangeably in conversation but are distinct in the model. **Athlete** = the person/user; **Trainee** = the coach–athlete relationship record. Use **Athlete** when referring to a person, **Trainee** when referring to the relationship or its billing/cost data.
- **"Plan"** is ambiguous: it appears as `Plan` (database model), **Workout Plan** (AI-generated proposal), and **Platform Subscription plan** (coach billing tier). Prefer **Workout Plan** for AI output and **Platform Subscription** for the coach's own SaaS tier.
- **"Transaction"** could mean either a **Trainee Transaction** (coach-to-athlete payment record) or a **Credit Ledger** entry (coach platform credit change). Always qualify: **Trainee Transaction** or **Credit Ledger entry**.
